## 1. MLX Metal Kernel Anatomy

### 1.1 GEMV

Fact (C:0.95): MLX routes general matmul to GEMV whenever `std::min(M, N) == 1`, otherwise to steel GEMM [`matmul.cpp#L1281-L1307`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/matmul.cpp#L1281-L1307). AddMM-shaped vector cases route to `gemv_axbpy`, fusing the matmul with an alpha/beta/bias epilogue [`matmul.cpp#L1413-L1435`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/matmul.cpp#L1413-L1435).

Fact (C:0.95): The GEMV kernel decomposes an `(M=out_vec_size, K=in_vec_size)` matrix into `(blockM, blockN)` regions across threadgroups. Each thread works on `(TM,TN)` tiles, loads `TN` matrix/vector elements, locally accumulates, and threadgroup reductions produce `blockM` outputs [`gemv.metal#L52-L73`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/gemv.metal#L52-L73).

Fact (C:0.95): The per-thread load path is vectorized by unrolled `TN` element loads into thread arrays. The main loop loads the vector coefficient block, loads corresponding matrix row chunks, multiplies, and accumulates in thread-local `result[TM]` registers [`gemv.metal#L75-L103`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/gemv.metal#L75-L103), [`gemv.metal#L159-L197`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/gemv.metal#L159-L197).

Fact (C:0.95): The first reduction uses SIMD-group shuffle via `simd_shuffle_down`; if `BN > 1`, threadgroup memory plus `threadgroup_barrier` combines SIMD-group partials [`gemv.metal#L199-L229`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/gemv.metal#L199-L229). Output write optionally applies the fused axpby epilogue [`gemv.metal#L231-L243`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/gemv.metal#L231-L243).

Fact (C:0.95): Host-side dispatch selects tile parameters from shape and transpose state, constructs a shape-specialized kernel name, launches with threadgroup size `(32,bn,bm)`, and dispatches `n_tgp` output blocks over batch [`matmul.cpp#L1042-L1179`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/matmul.cpp#L1042-L1179). The shader instantiates multiple `BM/BN/SM/SN/TM/TN` combinations for float32, float16, bfloat16, and complex64 [`gemv.metal#L448-L546`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/gemv.metal#L448-L546).

Fact (C:0.9): MLX also has masked GEMV variants that keep mask checks in the vector kernel [`gemv_masked.h#L190-L255`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/gemv_masked.h#L190-L255), with masked/transposed kernel instantiations [`gemv_masked.metal#L11-L76`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/gemv_masked.metal#L11-L76).

### 1.2 Steel GEMM: Tiling, `simdgroup_matrix`, and Threadgroup Staging

Fact (C:0.95): MLX routes non-vector matmul through steel GEMM and selects tile parameters based on device class, dtype, transpose case, and problem shape [`matmul.cpp#L88-L169`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/matmul.cpp#L88-L169). NAX fused GEMM uses larger tuned tiles, function constants for batch/epilogue/alignment, tile swizzling, threadgroup size `(32,wn,wm)`, and one dispatch over `(tiles_n, tiles_m, batch)` [`matmul.cpp#L175-L338`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/matmul.cpp#L175-L338).

Fact (C:0.95): The core steel GEMM kernel allocates padded threadgroup memory for A and B tiles, defines block loaders for A/B, and defines a `BlockMMA` object for simdgroup matrix multiply-accumulate. The main loop barriers, loads A/B tile blocks into threadgroup memory, barriers again, calls `mma_op.mma(As, Bs)`, and advances loaders [`gemm.h#L37-L120`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/gemm/gemm.h#L37-L120).

Fact (C:0.95): Steel block loaders use an aligned `ReadVector` type and copy vector chunks from device memory into threadgroup memory; safe loads use bounds checks and zero-fill invalid tile positions [`loader.h#L38-L128`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/gemm/loader.h#L38-L128).

Fact (C:0.95): Steel MMA includes `<metal_simdgroup_matrix>`, constrains fragments to 8x8, maps each SIMD-group lane to fragment elements, and calls `simdgroup_multiply_accumulate` [`mma.h#L1-L55`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/gemm/mma.h#L1-L55), [`mma.h#L181-L206`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/gemm/mma.h#L181-L206). Apple documents `simdgroup_matrix<T,Cols,Rows>` and `simdgroup_multiply_accumulate` returning `d = a * b + c` in the [Metal Shading Language Specification](https://developer.apple.com/metal/Metal-Shading-Language-Specification.pdf).

Fact (C:0.95): `BlockMMA` iterates over `BK` in 8-element fragments, applies simdgroup barriers around tile loads, performs tiled matrix multiply-accumulate, and stores the accumulated tile to device memory [`mma.h#L448-L546`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/gemm/mma.h#L448-L546).

Fact (C:0.95): MLX has split-K routes for small MxN/large K and NAX large-K cases [`matmul.cpp#L921-L967`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/matmul.cpp#L921-L967). Regular split-K writes partials to `C_split` then launches an accumulation kernel [`matmul.cpp#L500-L648`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/matmul.cpp#L500-L648). The NAX split-K host path uses a 1D K-partition-major grid to prevent driver/HW heuristics from launching threadgroups in non-linear order [`matmul.cpp#L776-L782`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/matmul.cpp#L776-L782) — this is a documented MLX source comment, not an external scheduling claim.

### 1.3 SDPA / Attention

Fact (C:0.95): MLX has two Metal attention families. Vector SDPA handles decode/small-query cases; steel full attention handles longer query sequences. Vector SDPA applies when `q_len <= 8`, `q_len <= k_len`, head dim equals value dim in {64, 96, 128, 256}, and `q_len * gqa_factor <= 32`; full steel attention supports `q_len > 8` with dims {64, 80, 128} [`scaled_dot_product_attention.cpp#L591-L640`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/scaled_dot_product_attention.cpp#L591-L640).

Fact (C:0.95): Vector SDPA dispatch uses one kernel over `(B*H, q_len, 1)` with threadgroup size `(1024,1,1)` and function constants for mask type, causal mode, and query layout [`scaled_dot_product_attention.cpp#L329-L416`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/scaled_dot_product_attention.cpp#L329-L416). Instantiations cover head dims 64, 96, 128, and 256 for float, bfloat16, and float16 [`scaled_dot_product_attention.metal#L9-L43`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/scaled_dot_product_attention.metal#L9-L43).

Fact (C:0.95): The vector SDPA shader reads one scaled query into registers, loops over keys, computes a dot product with `simd_sum`, applies optional mask/causal conditions, updates online-softmax max/sum accumulators, and updates the output accumulator in the same loop [`sdpa_vector.h#L83-L147`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/sdpa_vector.h#L83-L147). After the loop it combines partial max/sum/output via threadgroup/SIMD reductions and writes the normalized output directly [`sdpa_vector.h#L149-L176`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/sdpa_vector.h#L149-L176).

Fact (C:0.95): The vector path does not materialize a score matrix in the kernel source; scores are scalar/thread-local values inside the key loop [`sdpa_vector.h#L98-L147`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/sdpa_vector.h#L98-L147). FlashAttention-2 identifies standard attention's score/probability materialization to HBM as the O(N²) memory problem and describes tiled online-softmax attention that updates output without writing S/P to HBM [`FA2`](https://arxiv.org/abs/2307.08691).

Fact (C:0.95): MLX routes vector SDPA to a two-pass fused attention path on large devices for long KV (`k.shape(2) >= 1024` on max/ultra) or GQA long KV (`k.shape(2) >= 4096`) [`scaled_dot_product_attention.cpp#L743-L753`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/scaled_dot_product_attention.cpp#L743-L753). The two-pass path picks a block count based on architecture, KV length, and SIMD groups, writes partial outputs/sums/maxs, then runs a second reduction pass [`scaled_dot_product_attention.cpp#L418-L587`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/scaled_dot_product_attention.cpp#L418-L587).

Fact (C:0.95): Full steel attention stages Q and KV tiles in threadgroup memory, with K transposed for QK [`steel_attention.h#L110-L180`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/attn/kernels/steel_attention.h#L110-L180). It loops over KV blocks, loads K, computes `S = Q @ K.T`, applies scale and masks, loads V, updates row max/sum via online softmax, updates O via MMA against V, normalizes, and stores [`steel_attention.h#L249-L476`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/attn/kernels/steel_attention.h#L249-L476).

Inference (C:0.8): MLX's SDPA implementation is flash-style in the narrow structural sense: tiling/streaming, online max/sum softmax, and direct output accumulation without a materialized score/probability matrix. It should not be described as a proven FA2 implementation unless a source beyond these files establishes that equivalence.

### 1.4 Kernel Fusion Patterns

Fact (C:0.95): GEMV fuses the vector matmul and axpby-style epilogue when `kDoAxpby` is enabled [`gemv.metal#L231-L243`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/gemv.metal#L231-L243); host AddMM vector paths route to `gemv_axbpy` [`matmul.cpp#L1413-L1435`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/matmul.cpp#L1413-L1435).

Fact (C:0.95): Fused steel GEMM supports optional source C and alpha/beta epilogues through function constants inside the GEMM kernel [`steel_gemm_fused.h#L9-L41`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/gemm/kernels/steel_gemm_fused.h#L9-L41), [`steel_gemm_fused.h#L165-L204`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/gemm/kernels/steel_gemm_fused.h#L165-L204).

Fact (C:0.95): Full steel attention fuses QK, scale, causal/explicit mask application, softmax normalization state, V multiplication, and final O store within one kernel [`steel_attention.h#L249-L476`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/attn/kernels/steel_attention.h#L249-L476). Vector SDPA fuses dot, mask/causal check, softmax update, and output accumulation in one kernel [`sdpa_vector.h#L83-L176`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/sdpa_vector.h#L83-L176).

Inference (C:0.75): MLX's practical fusion is selective around bandwidth-sensitive boundaries: GEMV/AddMM epilogues, GEMM/AddMM epilogues, attention score/softmax/value, and mask handling. Split-K remains multi-dispatch because partial accumulation requires it [`matmul.cpp#L500-L648`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/matmul.cpp#L500-L648).

### 1.5 Dispatch, Command-Buffer Batching, and Residency

Fact (C:0.95): MLX's `CommandEncoder` tracks input/output buffers and only inserts a Metal buffer memory barrier when an input was a previous output [`device.cpp#L300-L370`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/device.cpp#L300-L370). At encoder finalization, it uses fences to synchronize command encoders only when inputs overlap previous encoder outputs [`device.cpp#L376-L443`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/device.cpp#L376-L443).

Fact (C:0.95): MLX batches primitive work in command buffers. `eval()` commits only if `encoder.needs_commit()` is true; otherwise it leaves the command buffer open until stream finalization [`eval.cpp#L30-L84`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/eval.cpp#L30-L84). `needs_commit()` trips when op count exceeds `max_ops_per_buffer_` or tracked buffer MB exceeds `max_mb_per_buffer_` [`device.cpp#L445-L455`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/device.cpp#L445-L455).

Fact (C:0.95): MLX selects architecture-specific command-buffer thresholds: phone 20 ops/40 MB, base/pro 40 ops/40 MB, max/ultra 50 ops/50 MB, with environment overrides [`device.cpp#L482-L523`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/device.cpp#L482-L523). It creates compute encoders with `MTL::DispatchTypeConcurrent` [`device.cpp#L473-L479`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/device.cpp#L473-L479).

Fact (C:0.95): MLX builds a residency set when supported, calls `requestResidency()`, inserts its heap and non-heap buffers into the set, and attaches the set to the command queue [`resident.cpp#L8-L38`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/resident.cpp#L8-L38), [`allocator.cpp#L34-L66`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/allocator.cpp#L34-L66), [`device.cpp#L263-L279`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/device.cpp#L263-L279).

Fact (C:0.95): Apple documents that residency sets reduce CPU overhead versus per-encoder resource residency calls, can be attached to a command queue, automatically attach to command buffers at commit, and `requestResidency()` can reduce commit-to-GPU-start delay [Apple Residency Sets](https://developer.apple.com/documentation/metal/simplifying-gpu-resource-management-with-residency-sets).

## 2. Why MLX Resists Concurrent-Load Degradation (#77): Sourced Hypothesis

**Scope note**: The `~2.8%` concurrent-load degradation figure and the `25%` lattice figure are flow-provided benchmark context. `r1/landscape.md` explicitly reports that the MLX 2.8% figure was not found in MLX source or Apple docs. The mechanism below is a sourced hypothesis, not a proven causal explanation.

**Critical correction** (from critic review, confirmed against `e1/current_state.md`): Lattice Qwen3.5 decode **already** uses a single command buffer for all active layers with no intermediate waits. This is documented at `crates/inference/src/forward/metal_qwen35.rs:5971` ("Single command buffer for ALL 24 layers - no intermediate waits"), with one encoder at `:5978`, one submit at `:6021`, and one commit/wait at `:6023`. The same single-submit pattern holds for prefill (`metal_qwen35.rs:6311`, `:6312`, `:7091`) and batch verification (`metal_qwen35.rs:4792`, `:5407`). The #77 gap is **not** explained by per-layer command buffer submission; that architecture is already in place.

### 2.1 Queue-Level Residency Sets (Inference, C:0.8)

MLX inserts its heap/buffers into a residency set, calls `requestResidency()`, and attaches the set to the command queue [`resident.cpp#L8-L38`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/resident.cpp#L8-L38), [`allocator.cpp#L34-L66`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/allocator.cpp#L34-L66), [`device.cpp#L263-L279`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/device.cpp#L263-L279). Apple states residency sets mitigate per-encoder CPU overhead, queue attachment is more efficient than per-command-buffer attachment, and `requestResidency()` can reduce commit-to-GPU-start delay [Apple Residency Sets](https://developer.apple.com/documentation/metal/simplifying-gpu-resource-management-with-residency-sets). Under concurrent GPU load, this commit-to-start delay reduction is a plausible differentiator. `e1/current_state.md` does not identify a comparable residency-set mechanism in lattice.

This inference exceeds direct source evidence that queue-level residency explains the 25% vs 2.8% gap. **A concurrent-load benchmark gate with Metal System Trace counters is required before attributing the gap to this mechanism.**

### 2.2 Op/MB Commit Thresholds Across Primitive Evals (Inference, C:0.8)

MLX leaves command buffers open across primitive evals until an op-count or MB threshold triggers a commit; max/ultra defaults are 50 ops or 50 MB per command buffer [`eval.cpp#L30-L84`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/eval.cpp#L30-L84), [`device.cpp#L445-L523`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/device.cpp#L445-L523). Apple documents that a compute encoder can encode multiple kernel dispatch commands into one command buffer [`MTLComputeCommandEncoder`](https://developer.apple.com/documentation/metal/mtlcomputecommandencoder). This threshold-governed cross-eval batching differs from lattice's explicit per-path structure and may reduce CPU/driver overhead under concurrent load by coalescing more work before submission.

### 2.3 Kernel Occupancy and Memory Intensity (Inference, C:0.75)

MLX routes decode-like attention to vector SDPA for `q_len <= 8` [`scaled_dot_product_attention.cpp#L591-L640`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/scaled_dot_product_attention.cpp#L591-L640) and routes general and quantized GEMM through steel/BlockMMA-style kernels [`gemm.h#L37-L120`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/gemm/gemm.h#L37-L120), [`fp_quantized.h#L630-L725`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/fp_quantized.h#L630-L725). Lattice's general FP GEMM path is scalar 16x16 tiled (`metal.rs:1537`, `:1552`; `metal_gemm.rs:31`, `:61`) and Q4 batch GEMM falls back to scalar outside Apple9 (`metal_qwen35.rs:1527`, `:1528`, `:9888`). Under competing GPU load, lower-occupancy or more memory-intensive kernels lose more wall-clock time.

### 2.4 Dependency-Tracked Barrier Minimization (Inference, C:0.65)

MLX inserts a buffer barrier only when an input was a previous output [`device.cpp#L300-L443`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/device.cpp#L300-L443). This preserves more scheduling freedom within a command buffer. The scheduling effect is not directly documented by Apple or MLX in the cited sources.

### 2.5 Validation Requirement

A concurrent-load benchmark with Metal System Trace counters — commit-to-start delay, residency events, GPU occupancy, command-buffer count, bytes moved — is required before attributing the 25% vs 2.8% gap to any specific mechanism. The #77 roadmap item (Rank 2, §4) must gate on this benchmark. The 2.8% MLX figure remains unverified primary-source claim in this study.

## 3. Gap Analysis: Lattice vs MLX (file:line from `e1/current_state.md`)

### G1. Attention / Flash (Issues #13, #126)

**Lattice decode**: `metal_qwen35.rs:418-419` — flash-style GQA decode, single-pass online softmax, one threadgroup per KV head for `cache_len <= 512`. `metal_qwen35.rs:562,565,689,702,10056-10069` — partitioned partial/reduce kernels for `cache_len > 512`, writing `(m,l,acc)` state, no global score matrix. `metal_qwen35.rs:452,518,543` — threadgroup-local scores, not a global `[heads, seq, seq]` buffer.

**Lattice Qwen3 embedding (active path)**: `metal.rs:492-493,1398,1626,1672` — dispatches `dispatch_fused_attention` with no global scores buffer, eliminates 3 separate dispatches.

**Legacy dead code** (Qwen3 embedding only, NOT Qwen3.5): `metal.rs:140-142,181,187,238` — `attn_scores`/`attn_softmax`/`attn_context` kernels exist in `metal.rs` source but the active forward path uses fused attention. These are maintenance drift, not active compute cost. They are **not present** in `metal_qwen35.rs`.

**MLX**: Vector SDPA for `q_len <= 8`: `(B*H, q_len, 1)` grid, 1024-thread group, `simd_sum` dot, online softmax, direct output accumulation [`sdpa_vector.h#L83-L176`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/sdpa_vector.h#L83-L176). Full steel attention for `q_len > 8`: tiled Q/K/V staging, online softmax, no materialized score matrix [`steel_attention.h#L249-L476`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/attn/kernels/steel_attention.h#L249-L476). Two-pass long-KV vector path on large devices [`scaled_dot_product_attention.cpp#L418-L587`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/scaled_dot_product_attention.cpp#L418-L587). FlashAttention-2 establishes the theoretical basis: tiled online softmax eliminates the O(N²) memory bottleneck from score materialization [`FA2`](https://arxiv.org/abs/2307.08691).

**Divergence**: The gap is routing breadth and prefill coverage. Lattice has flash-style mechanics in Qwen3.5 decode, so #13 is not a total absence. MLX has an explicit `q_len <= 8` vector SDPA route plus a full-query tiled attention path for longer sequences; lattice's prefill paths are not shown in e1 as using an equivalent full tiled online-softmax path.

**Impact**: High for #126/#13 when prefill or short multi-token verification uses paths outside Qwen3.5 decode flash. Medium for #77 because online-softmax decode already removes the worst global-score traffic.

### G2. GEMM / Steel Tiling (Issue #126)

**Lattice**: `metal_gemm.rs:31,45,61` — 16x16 tile, stages `tA`/`tB` in threadgroup memory, scalar inner products — no `simdgroup_matrix`. `metal_gemm.rs:198,201` — dispatches 16x16 threadgroups then **commits and waits per helper call**. `metal.rs:1537,1552` — Qwen3 embedding uses basic 16x16 for all shapes because rectangular variants had poor Metal 4 occupancy. `metal_qwen35.rs:2229,2289,2296` — `simdgroup_multiply_accumulate` only in optional Q4 tiled Apple9 path, absent from main FP16/BF16 GEMM.

**MLX**: Device/shape-specific tile selection [`matmul.cpp#L88-L169`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/matmul.cpp#L88-L169), aligned vectorized loaders into padded threadgroup memory [`loader.h#L38-L128`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/gemm/loader.h#L38-L128), `simdgroup_multiply_accumulate` MMA throughout [`mma.h#L181-L206`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/gemm/mma.h#L181-L206), split-K for small MxN/large K [`matmul.cpp#L921-L967`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/matmul.cpp#L921-L967).

**Impact**: High for #126 prefill and batch verification. Scalar-thread 16x16 GEMM is the clearest structural divergence from MLX steel.

### G3. Command-Buffer / Runtime (Issue #77)

**Lattice main paths** (already single-submit, confirmed from e1): `metal_qwen35.rs:5971,5978,6021,6023` — Qwen3.5 decode: single command buffer for all layers, one encoder, one submit, one wait. `metal_qwen35.rs:6311,6312,7091` — prefill: same pattern. `metal_qwen35.rs:4792,5407` — batch verification: same pattern. `metal_qwen35.rs:9704` — top-k appended to same encoder as logits.

**Remaining lattice edge waits**: `metal_gemm.rs:198,201` — standalone GEMM helper commits and waits per call. `metal_qwen35.rs:4556,4595` — checkpoint save/restore blit command buffers wait. `metal_qwen35.rs:6124` — CPU logits readback after GPU completion.

**Lattice residency**: Not documented in e1 — no queue-level residency set mechanism found.

**MLX**: Op/MB threshold batching [`device.cpp#L445-L523`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/device.cpp#L445-L523), dependency-tracked barriers only [`device.cpp#L300-L443`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/device.cpp#L300-L443), queue-level residency sets [`resident.cpp#L8-L38`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/resident.cpp#L8-L38), `MTL::DispatchTypeConcurrent` encoders [`device.cpp#L473-L479`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/device.cpp#L473-L479).

**Impact**: High for #77, but mechanistically uncertain. The largest submission granularity issue is already fixed in the main Qwen3.5 path. Remaining divergence is residency setup, edge-path waits, and kernel-level occupancy.

### G4. Shader Organization (Issue #86)

**Lattice**: `metal_qwen35.rs:50,3607` — all Qwen3.5 kernels in one inline Rust `MSL_SOURCE` string, compiled with `new_library_with_source`. `metal_qwen35.rs:2226,3620` — optional Q4 in second `concat!` string, compiled only for Apple9. `metal.rs:17,1048,1136` — Qwen3 embedding in inline `MSL_TEMPLATE` with runtime constant substitution. `metal_gemm.rs:27,123` — separate inline `SHADER_SOURCE`. Duplicated GEMM families: `metal_qwen35.rs:56` (`matmul_bt`), `metal.rs:23` (`matmul_bt`), `metal_gemm.rs:35` (`gemm_bt`).

**MLX**: Decomposed kernel families — GEMV in `gemv.metal` with instantiations [`gemv.metal#L448-L546`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/gemv.metal#L448-L546), steel GEMM decomposed into `gemm.h`/`loader.h`/`mma.h`/`steel_gemm_fused.metal` [`gemm.h#L37-L120`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/gemm/gemm.h#L37-L120), [`steel_gemm_fused.metal#L21-L33`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/steel/gemm/kernels/steel_gemm_fused.metal#L21-L33), attention in `sdpa_vector.h`/`steel_attention.h`/`.metal` files [`scaled_dot_product_attention.metal#L9-L43`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/scaled_dot_product_attention.metal#L9-L43).

**Impact**: Medium direct, high enabling. Inline duplication makes kernel variants difficult to diff, test, specialize, and reuse. Resolving #86 is a prerequisite for steel/flash/quantized roadmap items.

### G5. Quantized Matmul (Issue #120)

**Lattice**: Q8/Q4 decode GEMV with fused dequant and simd reductions — `metal_qwen35.rs:179,181,240,899,914,943,956`. Q8 batch GEMM with scalar accumulation + `simd_sum` — `metal_qwen35.rs:1455,1466,1504`. Q4 batch GEMM is a **scalar fallback cloned from Q8** (`metal_qwen35.rs:1527,1528`), dequantizing nibbles inline as `nibble * scale + bias` (`metal_qwen35.rs:1581`), reducing with `simd_sum` (`:1597`). Optional Apple9 tiled path at `metal_qwen35.rs:2229,2243,2253,2289,2296` using `simdgroup_float8x8`, `simdgroup_load`, and `simdgroup_multiply_accumulate`; dispatch falls back to scalar when absent (`metal_qwen35.rs:9875,9888`). Q4 storage uses mmap/no-copy with fused gate/up and qkvz layouts (`metal_qwen35.rs:2328,2365,11650,11715`).

**MLX**: Quantized block loader dequantizes packed weights into threadgroup memory [`fp_quantized.h#L157-L245`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/fp_quantized.h#L157-L245); steel `BlockMMA` plus quantized loaders for batch/prefill [`fp_quantized.h#L630-L725`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/kernels/fp_quantized.h#L630-L725); split-K quantized matmul capped by quantization groups [`quantized.cpp#L774-L860`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/quantized.cpp#L774-L860); group-size/bits specializations [`quantized.cpp#L720-L772`](https://github.com/ml-explore/mlx/blob/2e6632e5b84aa42d1bebecb0207092e223ff4d58/mlx/backend/metal/quantized.cpp#L720-L772).

**Impact**: Very high for #120 and #126 Q4/Q8 batch/prefill. Decode GEMV is already specialized; the structural gap is in batch/prefill quantized matmul falling back to scalar outside Apple9.
