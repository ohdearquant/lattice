## Context

Gemma 4 is Google's current model family (`google/gemma-4-*`), spanning E2B/E4B edge models through a
12B "Unified" (encoder-free) model to 26B-A4B MoE and 31B dense flagships. It is the second model
family this engine would support natively, alongside the Qwen3.5/3.6 hybrid GDN+GQA decoder
(`crates/inference/src/model/qwen35_config.rs`). Adding it is a new-family implementation, not a
config tweak: Gemma 4's attention, normalization, embeddings, FFN, and tokenizer are each
structurally different from Qwen3.5's, and it introduces two entirely new modality towers (vision,
audio) that must be built rather than adapted.

**Why E2B-it.** `google/gemma-4-E2B-it` is the smallest official Gemma 4 variant that is
end-to-end capable — text, image, and audio — in one checkpoint: Apache-2.0, public and
`gated: false` at audit (2026-07-14), 2.3B effective parameters (5.1B including embeddings). No
smaller vision-only fallback is needed because E2B already ships audio; the next-larger
audio-capable tier is E4B, and the next-larger vision-only tiers (26B-A4B MoE, 31B dense) are
strictly bigger and lack an audio path. The 12B "Unified" model uses a distinct encoder-free
`gemma4_unified` architecture and is not a substitute for an E2B implementation slice.

**Relationship to the Qwen vision lane.** ADR-069 recalibrated the Qwen3.5-0.8B vision path
(`crates/inference/src/vision/`) against its native `model.visual.*` ViT tower. That scaffold and
this ADR's Gemma vision/audio towers are **separate towers with separate weight schemas and separate
goldens** — they share the goldens-first discipline (HF reference outputs generated and checked
_before_ the corresponding lattice engine code is written, per this repo's differential-test-first
practice) and the same in-sequence placeholder-replacement pattern for injecting modality tokens, but
neither lane's tower, config parser, or Metal kernels are reused by the other. This ADR does not
modify `crates/inference/src/vision/` or ADR-069's scope; a later ADR would formalize any shared
abstraction once both towers exist and their actual commonality (not assumed commonality) is known.

## Decision

Build Gemma 4 E2B support as a ten-stage ladder. Every stage generates and checks its Hugging Face
golden **before** the corresponding lattice engine code is written (goldens-first, per this repo's
differential-test-first practice), and every stage adds a **fail-closed negative test** — a
deliberate mutation of the stage's own contract that must cause the stage's gate to fail, so the gate
is proven mutation-sensitive rather than decorative. Goldens and full checkpoints live in an external
asset store; only small expected IDs, tensor slices, and logits enter this repository as fixtures,
after a size/privacy review.

Stage 4 is the first architectural decision point requiring sign-off before dependent code lands
(cache ownership/indexing for the 20 shared-KV layers). Stages 5–6 (vision) and 7–9 (audio) are
independently verifiable and may proceed in either order once Stage 4 is complete; "E2B supported"
requires independent parity proof on all three — text-only, image+text, and audio+text — since a
passing text path does not validate vision (different embeddings/placeholders/preprocessor) and a
passing vision path does not validate audio (its own frontend, attention pattern, and length
contract).

## Risks

- **PLE + shared-KV/local-global cache-state interaction — highest silent-correctness risk.** It is
  easy to produce plausible-looking text with a wrong state owner or a wrong per-layer PLE residual
  and never reach HF parity by inspecting only final tokens. **Mitigation**: Stage 4's gate compares
  per-layer hidden states and the two shared-KV state families _before_ judging first-token parity,
  and its negative test deliberately mis-owns a shared-KV layer.
- **Tokenizer loader rejection.** The target's literal-space `Split` + `▁` normalizer is outside what
  `validate_bpe_pretokenizer` currently accepts, and is rejected by design rather than silently
  mis-tokenized. **Mitigation**: build an isolated Gemma tokenizer path (Stage 1) with its own
  goldens before any engine code depends on it; do not widen the general BPE loader's acceptance to
  make this pass.
- **Dual-RoPE correctness.** A per-layer-type RoPE dispatcher (local θ=1e4 standard vs global θ=1e6
  proportional over 25% of head dim) is new scope; this repo's RoPE-pairing bug history (interleaved
  vs stride-half convention mismatches surviving as multiply-inlined copies) means dispatcher bugs are
  plausible and easy to miss with only an aggregate-output check. **Mitigation**: differential-test
  each RoPE variant against the HF reference in isolation (per-layer-type golden) before wiring the
  full Stage-3/4 forward pass.
- **Audio frontend numerics.** The exact log-mel contract (semicausal pad, specific window/hop,
  HTK mel, `log(mel+0.001)`) is easy to approximate incorrectly with a generic STFT/mel
  implementation and still look "close." **Mitigation**: Stage 7's golden tolerance is checked across
  silence, tone, speech, short, and 30 s clips — not one canonical case — before any encoder code is
  written against the frontend's output.
- **Vision clip buffers treated as disposable.** `Gemma4ClippableLinear`'s persistent input/output
  min/max buffers are load-bearing when `use_clipped_linears=true`; skipping them as "just
  quantization metadata" would silently change vision-tower numerics. **Mitigation**: Stage 0's
  manifest explicitly accounts for these buffers (already reflected in the 658-tensor
  `model.vision_tower` count, G16), and Stage 5's per-block golden comparison would surface a mismatch
  if they were dropped.

## Consequences

**Positive.** The goldens-first, negative-test-per-stage ladder catches architecture and
cache-topology mismatches before engine code is written rather than after a plausible-but-wrong
model ships. The safetensors header extraction (G16) retires the family's earlier "no
index/manifest" blocker immediately, at zero weight-download cost, so Stage 0 can proceed today. A
second model family with a materially different attention/norm/FFN/tokenizer/modality contract is
also the forcing function that will show, empirically, which of the engine's Qwen-shaped
abstractions generalize and which are Qwen-specific — informing rather than pre-deciding any future
architecture-generalization work.

**Cost.** This is a large ladder: a wholly new tokenizer path, a new dual-RoPE dispatcher, new
sliding+global cache topology with 20-layer KV sharing, a new PLE mechanism, a new native vision
tower (not a SigLIP reuse), and an entirely new audio stack (log-mel frontend, Conv2d subsampler,
Conformer-like encoder). None of Qwen3.5's GDN attention code, shifted-RMSNorm, or SwiGLU path is
reusable for the text decoder; only lower-level kernel/cache-allocation discipline and safetensors
ingress plumbing carry over. Every stage requires its own committed golden fixtures and external-asset
provenance tracking.

**Risk.** If Stage 4's per-layer trace does not match before first-token comparison, the ladder
blocks there rather than shipping a model that produces plausible but wrong text — this is the
intended failure mode, not a defect in the plan. If the Gemma tokenizer path reveals that the
existing BPE loader's assumptions need to change more broadly than an isolated new path, that scope
change is escalated rather than absorbed silently into the general loader.

## Deferred / explicitly out of scope

- E4B, 12B "Unified" (distinct `gemma4_unified` architecture), 26B-A4B (MoE), and 31B variants — E2B
  is the sole target of this ADR.
- Video/frame input (the family card lists video-as-frames support on larger tiers; not in this
  ladder).
- Audio speech-out (TTS/neural-codec generation) — this ladder is speech-in (encoder) only, matching
  the audio-tensors lane's (#565) existing speech-in/speech-out split.
- Mixed image+audio interleaving in a single prompt — Stage 9 covers audio+text only; combined
  image+audio is a follow-up once both single-modality e2e stages are independently proven.
- Any Gemma-specific quantization tier (Q4/Q8/W3 analogues) — BF16 safetensors parity ships first;
  quantization is deferred until every tensor and clip buffer is audited and text/image/audio parity
  all hold, consistent with how this repo already sequences quantization work behind correctness
  (ADR-072).
- Metal kernels ahead of CPU parity at any stage — every modality's Metal path (Stages 4, 6, 9) is
  gated on its own CPU-parity stage passing first.
